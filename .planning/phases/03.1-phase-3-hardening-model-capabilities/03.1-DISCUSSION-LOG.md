## Model Catalog Depth

| Option | Description | Selected |
|--------|-------------|----------|
| Curated hardcoded catalog | Static data shipped with extension, works offline | |
| Dynamic API-fetched catalog | Call each provider's model list API on first use | ✓ |
| Hybrid | Hardcoded baseline + optional API refresh | |

**User's choice:** Dynamic API-fetched catalog
**Notes:** User wants latest models fetched from provider APIs, not stale hardcoded lists.

### Capability metadata approach

| Option | Description | Selected |
|--------|-------------|----------|
| Hardcoded metadata overlay | Static mapping of known models to capabilities, defaults for unknowns | ✓ |
| Provider API metadata only | Only what APIs return | |
| API + heuristic inference | Infer from model name patterns | |

**User's choice:** Hardcoded metadata overlay
**Notes:** Fetch model IDs from APIs, overlay with hardcoded capability data. User stressed that the overlay must include LATEST models from all major labs — not outdated examples.

### Which providers to fetch from

| Option | Description | Selected |
|--------|-------------|----------|
| Only configured providers | API key set + enabled | ✓ |
| All added providers | Even disabled ones | |
| All supported providers | All 8 regardless of config | |

**User's choice:** Only configured providers
**Notes:** No point calling APIs for unconfigured providers.

## Enabled Filtering Behavior

| Option | Description | Selected |
|--------|-------------|----------|
| Strict: error + clear message | Throw actionable error when all providers disabled | ✓ |
| Graceful: silent Nano fallback | Always fallback to Gemini Nano | |
| Warning + Nano fallback | Show warning but still try Nano | |

**User's choice:** Strict error with clear message
**Notes:** User must fix configuration before retrying. No silent fallbacks.

### Per-model enable/disable

| Option | Description | Selected |
|--------|-------------|----------|
| Yes, per-model toggle | Each model has its own enabled flag | ✓ |
| No, provider-level only | Disabling provider disables all models | |

**User's choice:** Per-model enable/disable
**Notes:** Fine-grained control — user can enable GPT-4o but disable GPT-3.5.

## SettingsManager API

| Option | Description | Selected |
|--------|-------------|----------|
| Individual getters/setters | Dedicated methods per field | ✓ |
| Bulk partial update | One `updatePreferences(partial)` method | |
| Minimal additions only | Only what's strictly missing | |

**User's choice:** Individual getters/setters
**Notes:** Clean API with dedicated methods like `setRoutingMode()`, `setFallbackChain()`, `addModel()`, etc.

### Input validation

| Option | Description | Selected |
|--------|-------------|----------|
| Validate in setters | SettingsManager validates all inputs | ✓ |
| Validate at route time only | Router validates when it uses the data | |

**User's choice:** Validate in setters
**Notes:** Setters reject invalid values immediately. Router can trust data from SettingsManager.
