# Phase 03.1: Phase 3 Hardening & Model Capabilities - Research

**Researched:** 2026-03-28
**Domain:** Chrome extension routing hardening, model catalog, enabled filtering, SettingsManager API
**Confidence:** HIGH

## Summary

Phase 03.1 is a hardening pass on the Phase 3 routing infrastructure. The existing ProviderRouter never checks `config.enabled` on providers and has no concept of per-model enable/disable. SettingsManager is missing granular setters for routingMode, fallbackChain, triggerWords, and all model CRUD operations. The model sheet starts empty with no seeding logic. This phase fills these gaps by: (1) adding strict enabled filtering at both provider and model levels, (2) implementing dynamic API-fetched model catalog with hardcoded capability metadata overlay, (3) extending ModelCapabilities with new fields, (4) implementing all missing SettingsManager methods with input validation, and (5) achieving comprehensive test coverage for all routing paths.

**Primary recommendation:** Extend the existing `types.ts`, `settings-manager.ts`, and `provider-router.ts` in place. Create a new `model-catalog.ts` for the hardcoded overlay + API fetch logic. Add validation using the existing `zod` dependency. Follow the established test mock patterns from `tests/settings-manager.test.ts` and `tests/provider-router.test.ts`.

### Pattern 1: Enabled Filtering in Route Flow
**What:** ProviderRouter checks both provider-level and model-level enabled status before considering a provider for routing.
**When to use:** Every routeCapability() call.
**Example:**
```typescript
// In provider-router.ts routeCapability()
// After getting model sheet, before heuristic scoring:
for (const [providerId, entry] of Object.entries(modelSheet)) {
  // Skip disabled models
  if (entry.enabled === false) continue;

  // Also check provider-level enabled
  const config = await configManager.getProvider(providerId);
  if (!config || !config.enabled) continue;

  // ... existing scoring logic
}
```

### Pattern 2: Zod Schema for Setter Validation
**What:** Use Zod schemas to validate inputs in SettingsManager setters, producing specific error messages.
**When to use:** All setters that accept user input (setRoutingMode, setFallbackChain, addTriggerWord, addModel).
**Example:**
```typescript
// In settings-manager.ts
import { z } from 'zod';

const RoutingModeSchema = z.enum(['auto', 'manual'], {
  errorMap: () => ({ message: "routingMode must be 'auto' or 'manual'" })
});

async setRoutingMode(mode: string): Promise<void> {
  const validated = RoutingModeSchema.parse(mode);
  const prefs = await this.getRoutingPreferences();
  prefs.routingMode = validated;
  await chrome.storage.local.set({ [PREFS_KEY]: prefs });
}
```

### Pattern 3: Model Catalog Seeding
**What:** On first access of model sheet, if empty, trigger API fetch from all enabled+configured providers, then merge with hardcoded overlay.
**When to use:** `refreshModelCatalog()` and initial model sheet access.
**Example:**
```typescript
// In model-catalog.ts
async seedModelCatalog(configManager: ProviderConfigManager): Promise<void> {
  const providers = await configManager.listProviders();
  const enabledWithKeys = providers.filter(p => p.enabled && p.apiKeyId);

  for (const provider of enabledWithKeys) {
    const models = await fetchModelsFromProvider(provider);
    // Merge each model with hardcoded overlay
    for (const model of models) {
      const overlay = HARDCODED_CAPABILITIES[model.id];
      const entry: ModelSheetEntry = {
        modelId: model.id,
        providerType: provider.type,
        enabled: true, // Default enabled
        capabilities: overlay ?? DEFAULT_CAPABILITIES,
        tier: overlay?.tier ?? inferTier(overlay),
      };
      // Add to model sheet
    }
  }
}
```

### Anti-Patterns to Avoid
- **Scoring disabled entries:** Never compute heuristic scores for disabled providers/models. Skip them entirely before the scoring loop. (D-11)
- **Generic validation errors:** Never throw "Invalid input". Always include the field name, the invalid value, and what was expected. (D-15)
- **Mutating model sheet without validation:** Every write to chrome.storage.local should be preceded by input validation that rejects bad data before it reaches storage.
- **Synchronous model catalog fetch:** Model catalog seeding must be async. The model sheet starts empty and populates on first access -- callers must await.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Runtime input validation | Manual if/else chains per setter | `zod` schemas with custom errorMap | Zod already installed; provides consistent error messages, type inference, and composability |
| Model list API calls | Raw fetch() with manual auth header construction | Existing adapter instances via ProviderFactory | Adapters already handle API key resolution, error handling, and provider-specific auth |
| String enum validation | `if (mode !== 'auto' && mode !== 'manual')` | `z.enum(['auto', 'manual'])` | Zod gives better error messages and is extensible |
| Model capability defaults | Scattered ternary operators | Centralized `DEFAULT_CAPABILITIES` constant | One place to change defaults for unknown models |

**Key insight:** The project already has all the building blocks -- adapters for API calls, zod for validation, chrome.storage.local for persistence. This phase wires them together; no new libraries needed.

## Common Pitfalls

### Pitfall 1: Provider ID vs Model Sheet Key Confusion
**What goes wrong:** The current code uses provider IDs as keys in the model sheet (`modelSheet[providerId]`), but after dynamic catalog fetch, one provider may have multiple models. The data model must change from `Record<providerId, ModelSheetEntry>` to `Record<modelId, ModelSheetEntry>` where each entry also stores its parent `providerId`.
**Why it happens:** Phase 3 used a 1:1 provider-to-model mapping. Dynamic catalog creates 1:many.
**How to avoid:** Change model sheet key to be the model ID (not provider ID). Each `ModelSheetEntry` already has `providerType` -- add a `providerId` field to link back to the provider config.
**Warning signs:** Tests that index model sheet by provider ID instead of model ID.

### Pitfall 2: Model Sheet Key Migration
**What goes wrong:** Existing model sheet data uses provider IDs as keys. After the key change, old data becomes inaccessible.
**Why it happens:** chrome.storage.local persists across extension updates.
**How to avoid:** Add a migration check in `getModelSheet()` -- if any key looks like a provider ID (starts with `provider_`), re-key the entries using modelId. Or clear and re-seed if data is from Phase 3.
**Warning signs:** Empty model sheet after upgrade, or entries indexed by wrong keys.

### Pitfall 3: Heuristic Scoring Disabled Models
**What goes wrong:** The scoring loop in `routeCapability()` iterates all model sheet entries without checking `enabled`. A disabled model could get the highest score and be selected.
**Why it happens:** The current code has no `enabled` field on ModelSheetEntry.
**How to avoid:** Add `entry.enabled !== false` check at the start of the scoring loop, before any score computation.
**Warning signs:** Tests pass but routing picks disabled providers.

### Pitfall 4: API Fetch Rate Limiting
**What goes wrong:** Calling every provider's model list API on first use could hit rate limits, especially for free-tier keys.
**Why it happens:** No caching or rate limiting on catalog fetch.
**How to avoid:** Cache the fetched model list with a TTL (Claude's discretion). Only re-fetch on explicit `refreshModelCatalog()` call. The initial seed is a one-time operation per provider.
**Warning signs:** 429 errors during model catalog seeding.

### Pitfall 5: Anthropic Has No `/v1/models` Endpoint
**What goes wrong:** Code assumes all providers have a model list API like OpenAI's `/v1/models`. Anthropic does not expose this endpoint.
**Why it happens:** Anthropic's API design philosophy differs from OpenAI-compatible providers.
**How to avoid:** For Anthropic, use the hardcoded overlay data exclusively -- do not attempt API fetch. The overlay already contains all known Anthropic models with their capabilities. Flag this in the catalog logic per provider type.
**Warning signs:** 404 errors when fetching Anthropic model list.

### Pitfall 6: Embedding Provider Switching Without Warning
**What goes wrong:** The existing `EmbeddingProviderSwitchError` guard in `setCapabilityProvider` works, but `setFallbackChain` does not validate that embedding providers are being added to the fallback chain for non-embedding capabilities.
**Why it happens:** Fallback chain is capability-agnostic in the current data model.
**How to avoid:** `setFallbackChain` should validate that entries reference existing providers (D-14). The embedding warning is already handled by `setCapabilityProvider`.
**Warning signs:** Fallback chain references deleted or non-existent provider IDs.

### Pitfall 7: Gemini Nano in Model Sheet
**What goes wrong:** Gemini Nano has no API key and no model list API. If the catalog seeding logic tries to fetch models from Gemini Nano, it will fail.
**Why it happens:** Gemini Nano is a local provider with no cloud API.
**How to avoid:** Skip Gemini Nano in the API fetch loop. Hardcode Gemini Nano's entry in the overlay with `providerType: 'gemini-nano'`, cost: 'free', quality: 'basic'.
**Warning signs:** Errors during catalog seeding related to Gemini Nano.

### SettingsManager Validation Pattern (D-14, D-15)
```typescript
// In settings-manager.ts
import { z } from 'zod';

const RoutingModeSchema = z.enum(['auto', 'manual'], {
  errorMap: () => ({ message: "routingMode must be 'auto' or 'manual'" }),
});

async setRoutingMode(mode: unknown): Promise<void> {
  const validated = RoutingModeSchema.parse(mode); // Throws with specific message
  const prefs = await this.getRoutingPreferences();
  prefs.routingMode = validated;
  await chrome.storage.local.set({ [PREFS_KEY]: prefs });
}

async getRoutingMode(): Promise<'auto' | 'manual'> {
  const prefs = await this.getRoutingPreferences();
  return prefs.routingMode;
}

async setFallbackChain(chain: string[]): Promise<void> {
  if (!Array.isArray(chain)) {
    throw new Error("fallbackChain must be an array of provider ID strings");
  }
  // D-14: Validate entries reference existing enabled providers
  const configManager = getProviderConfigManager();
  const providers = await configManager.listProviders();
  const enabledIds = new Set(providers.filter(p => p.enabled).map(p => p.id));
  const invalid = chain.filter(id => !enabledIds.has(id));
  if (invalid.length > 0) {
    throw new Error(
      `Fallback chain contains non-existent or disabled providers: ${invalid.join(', ')}. ` +
      `Remove them or enable the providers first.`
    );
  }
  const prefs = await this.getRoutingPreferences();
  prefs.fallbackChain = chain;
  await chrome.storage.local.set({ [PREFS_KEY]: prefs });
}

async addTriggerWord(word: string, providerId: string): Promise<void> {
  if (!word || typeof word !== 'string') {
    throw new Error("Trigger word must be a non-empty string");
  }
  if (!providerId || typeof providerId !== 'string') {
    throw new Error("providerId must be a non-empty string referencing an existing provider");
  }
  const prefs = await this.getRoutingPreferences();
  prefs.triggerWords = { ...prefs.triggerWords, [word]: providerId };
  await chrome.storage.local.set({ [PREFS_KEY]: prefs });
}

async removeTriggerWord(word: string): Promise<void> {
  const prefs = await this.getRoutingPreferences();
  const { [word]: _, ...rest } = prefs.triggerWords;
  prefs.triggerWords = rest;
  await chrome.storage.local.set({ [PREFS_KEY]: prefs });
}
```

```typescript
      throw new Error(validation.error || 'Connection validation failed');
    }
    return adapter;
  } catch (e: any) {
    diagnostics.push({ providerId, error: e.message || 'Unknown error' });
  }
}

// D-10: Clear actionable error when all filtered out
throw new Error(
  `No enabled providers available for '${capability}' capability. ` +
  `Enable a provider and its models in settings. ` +
  `Diagnostics: ${JSON.stringify(diagnostics)}`
);
```

## Open Questions

1. **Model Sheet Key Migration Strategy**
   - What we know: Current model sheet uses provider IDs as keys. New model sheet should use model IDs.
   - What's unclear: Whether any production data exists that needs migration (Phase 3 was just completed).
   - Recommendation: Since Phase 3 was just completed and no users have persisted data, clear the model sheet on first load after this phase and re-seed. Add a version field to detect stale format.

2. **Anthropic Model List API**
   - What we know: Anthropic does not expose a `/v1/models` endpoint.
   - What's unclear: Whether Anthropic plans to add one.
   - Recommendation: Use hardcoded overlay for all Anthropic models. No API fetch attempt for `providerType === 'anthropic'`. This is the simplest correct approach.

3. **Google Gemini Model List API**
   - What we know: Google has a models.list endpoint in the Generative AI API, but it may not be accessible via the Vercel AI SDK `@ai-sdk/google` provider.
   - What's unclear: The exact URL pattern and authentication method for listing Google models.
   - Recommendation: Use hardcoded overlay for Google models initially. If needed, the `@google/generative-ai` SDK (already installed at v0.24.1) has a `listModels()` method that could be used as a fallback.
