# Phase 03.1: Phase 3 Hardening & Model Capabilities - Context

**Gathered:** 2026-03-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Harden Phase 3 routing system — enabled provider/model filtering, predefined model catalog with extended capabilities, missing SettingsManager methods, intent-aware heuristics validation, and comprehensive test coverage. This is a hardening pass that fills gaps in the existing routing infrastructure without adding new capabilities.

**What's IN scope:**
- Enabled provider + model filtering in ProviderRouter
- Dynamic API-fetched model catalog from configured providers
- Hardcoded capability metadata overlay (context window, max output tokens, image/video/audio analysis)
- Per-model enable/disable toggle
- Missing SettingsManager methods (routingMode, fallbackChain, triggerWords, model CRUD)
- Input validation in SettingsManager setters
- Clear error messages for misconfigured capabilities
- Test coverage for all routing paths (trigger words, embeddings, speech, enabled filtering)

**What's OUT of scope:**
- Settings UI (Phase 6)
- Speech-to-text integration (Phase 4+5)
- Additional provider adapters (Phase 4)
- HybridAIEngine integration (Phase 5)

</domain>

<decisions>
## Implementation Decisions

### Model Catalog
- **D-01:** Dynamic API-fetched model catalog — call each configured provider's model list API (e.g., OpenAI `/v1/models`, Anthropic `/v1/models`) to populate the model sheet
- **D-02:** Fetch only from providers that have an API key configured AND are enabled — skip unconfigured providers entirely
- **D-03:** Hardcoded capability metadata overlay — maintain a static mapping of known models to their capabilities (context window, max output tokens, supportsImageAnalysis, supportsVideoAnalysis, supportsAudioAnalysis)
- **D-04:** Overlay must include the latest models from all major labs/providers — researcher must fetch current model data at execution time (not rely on stale examples)
- **D-05:** Unknown/unrecognized models (not in overlay) get sensible defaults: 4096 context window, 2048 max output, no image/video/audio, basic quality tier
- **D-06:** Model catalog seeds on first use — when model sheet is empty, trigger API fetch + overlay merge

### Model Capabilities Extension
- **D-07:** Expand `ModelCapabilities` beyond current `supportsVision` + `contextWindow` to include: `maxOutputTokens`, `supportsImageAnalysis`, `supportsVideoAnalysis`, `supportsAudioAnalysis`
- **D-08:** Per-model enable/disable toggle — each model entry has its own `enabled` flag, separate from the provider-level toggle. Router only routes to enabled models from enabled providers.

### Enabled Filtering
- **D-09:** Strict filtering — ProviderRouter only considers providers that are both (a) provider-level enabled AND (b) have at least one enabled model
- **D-10:** When all providers for a capability are disabled or have no enabled models: throw a clear, actionable error (e.g., "No enabled providers for chat capability. Enable a provider in settings.")
- **D-11:** Heuristic scoring in auto mode must skip disabled providers and disabled models — no scoring computation for excluded entries

### SettingsManager API
- **D-12:** Individual getters/setters for each preference field: `getRoutingMode()`, `setRoutingMode(mode)`, `getFallbackChain()`, `setFallbackChain(chain)`, `getTriggerWords()`, `setTriggerWords(words)`, `addTriggerWord(word, providerId)`, `removeTriggerWord(word)`
- **D-13:** Model management methods: `addModel(providerId, entry)`, `removeModel(providerId)`, `getModel(providerId)`, `setModelEnabled(providerId, enabled)`, `refreshModelCatalog()` — triggers API fetch + overlay merge
- **D-14:** Input validation in setters — `setRoutingMode` validates 'auto' | 'manual', `setFallbackChain` validates entries reference existing enabled providers, `addModel` validates required fields, etc.
- **D-15:** Validation errors are specific and actionable (not just "Invalid input" but "routingMode must be 'auto' or 'manual'")

### Claude's Discretion
- Exact hardcoded overlay data (researcher will fetch latest model specs)
- Error message formatting and style
- Test mock strategy for provider APIs and Nano classifier
- Type definition field names for new capability flags
- File organization within `src/background/routing/`
- Caching strategy for API-fetched model lists

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **ProviderRouter.routeCapability():** Existing routing logic with heuristic scoring, trigger words, and fallback chain — needs enabled filtering added
- **SettingsManager:** Existing CRUD for preferences, model sheet, parameters — needs individual setters and validation
- **nano-classifier.ts:** Working Gemini Nano intent classifier — already produces complexity/intent/budget_signal
- **ProviderFactory.createAdapter():** Creates adapters by type — already async, handles key fetching
- **ProviderConfigManager:** Full provider CRUD with enabled flag per provider

### Established Patterns
- `chrome.storage.local` for all routing preferences (key `ai_pocket_routing_prefs`)
- Zod schema validation for structured AI output (nano-classifier)
- Singleton pattern for managers
- Type-only imports for verbatimModuleSyntax compliance (fixed in Phase 3 UAT)
- Error classes extending Error (EmbeddingProviderSwitchError pattern)

### Integration Points
- ProviderRouter reads from SettingsManager for routing decisions
- SettingsManager reads/writes chrome.storage.local directly
- ProviderFactory creates adapters from ProviderConfig (which has `enabled` field)
- Existing tests mock chrome.storage.local and ProviderFactory

### Gaps Identified
- `ModelCapabilities` only has `supportsVision` and `contextWindow` — needs 3-4 more fields
- `ProviderRouter` never checks `config.enabled` or filters models
- No model catalog seeding logic — model sheet starts as empty `{}`
- SettingsManager missing granular setters for routingMode, fallbackChain, triggerWords
- No test coverage for: trigger word routing, embeddings routing, speech routing, enabled filtering

</code_context>
